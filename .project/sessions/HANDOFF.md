## What Was Built (Session 16)

### SQLite → Postgres Migration (33 files)
- postgres.js tagged templates, tsvector + pgvector, all ops async
- Railway managed Postgres with automatic backups
- Deleted backup service (Railway handles backups)

### Autonomous Task Queue
- **Schema**: `scheduled_tasks` table with scheduling, provenance, 3P integration fields
- **Store**: Full CRUD + scheduler operations + approval + pin tracking (26 tests)
- **Tools**: list_tasks, create_task, update_task, delete_task, sync_tasks, delete_memory
- **Scheduler**: Polls every 60s, executes due tasks through agent pipeline, reschedules recurring
- **Executor**: Builds synthetic message → assembleContext → route → execute. Posts results to task channel.
- **Dedicated task channel**: Pinned messages = live dashboard. Channel messages = notifications. Threads = modifications + approvals.
- **Cross-channel management**: Create/modify/delete tasks from any channel. Thread changelog on cross-channel modifications.
- **Pin sync**: Startup + periodic (6h) + on-demand via sync_tasks. DB is source of truth.
- **Haiku pin summaries**: One-line summaries generated by Haiku, cached in `pin_summary` column.
- **Detail threads**: First thread reply has full task details. Edited when core details change.
- **Approval flow**: 👍 on pinned message approves pending tasks. Thread replies for discussion.
- **MCP integration**: Deep path has task tools via memory MCP server.

### Three-Tier Routing
- FAST (Sonnet): greetings, simple lookups, single-tool
- MEDIUM (Opus API): memory, task management, corrections, multi-tool reasoning
- DEEP (Opus CLI): cross-source research, document analysis
- Dramatically reduced hallucinations on memory-heavy conversations

### Bug Fixes & Improvements
- tsvector search: AND → OR for multi-word queries
- Message chunking: prevents msg_too_long
- Undelivered response fallback: saved to memory
- Error notification: always surface errors to user in Slack
- Preflight prompt: never imply starting without [PROCEED]
- Brain reaction: on user's message, not bot's confirmation
- Training wheels removed (was blocking delete_memory, delete_task)
- Memory delete tool added (soft-delete by ID)
- Owner-created tasks activate immediately (no approval needed)
- `delay` parameter for relative time scheduling ("2 minutes", "3 hours")
- `computeNextRun` supports "daily at 6am" time-of-day scheduling
