Device-Bound vs. Always Available

Claude Code is powerful but tethered to whatever terminal you're sitting at. The goal: make it accessible from anywhere via Slack, with persistent memory and proactive capabilities.

03 — Authentication

Max Plan OAuth on a Headless Server

The most critical design decision: using your Max subscription (not an API key) to power the bot. This keeps costs at zero beyond the VM.

Initial Setup (One-Time)
  1. SSH into your cloud VM
  2. Run claude in the terminal
  3. CLI outputs a URL — open it on your phone or laptop
  4. Approve the OAuth consent in your browser
  5. Credentials are stored in ~/.claude/ on the server
  6. All future Claude Code invocations use these credentials
Token Lifecycle

Max OAuth tokens expire periodically. The bot must handle this gracefully:

  1. Claude Code SDK call fails with auth error
  2. Bot catches the error, posts alert to Slack:
    "Auth expired — SSH in and re-run claude to re-authenticate"
  3. All queued tasks are paused until re-auth
  4. You SSH in, run the OAuth flow again (~30 seconds)
  5. Bot auto-resumes queued tasks
Why Not an API Key?

API keys charge per-token. For a personal always-on assistant with proactive tasks running throughout the day, this adds up fast. Your Max subscription gives you a flat monthly rate with generous usage limits — perfect for a single-user bot that runs many small tasks.

04 — Slack Bot Layer

The Interface

The Slack bot is the only user-facing surface. It receives your messages, delegates to Claude Code, and posts results back — handling all the formatting translation along the way.

Socket Mode

Bot connects outbound to Slack via WebSocket. No public IP, no domain, no SSL cert needed. Works with free Slack plans.

Thread Mapping

Each Slack thread ID maps to a Claude Code session ID. Replies in the same thread continue the same conversation with full context.

Response Handling

Posts a "working..." placeholder immediately. Replaces with result when done. Long responses (>3k chars) get uploaded as file snippets.

src/bot.ts — Core message handler
app.event('app_mention', async ({ event, say }) => {
  const thread_ts = event.thread_ts || event.ts;
  const sessionId = sessions.get(thread_ts);

  // Immediate feedback
  const thinking = await say({ text: 'Working on it...', thread_ts });

  // Queue with priority (interactive > scheduled)
  const result = await queue.add(async () => {
    return claude({
      prompt: event.text,
      workingDirectory: '/workspace',
      sessionId,
      systemPrompt: loadMemoryContext(),
    });
  }, { priority: 10 });

  // Persist session for thread continuity
  sessions.set(thread_ts, result.sessionId);

  // Format and post
  const text = markdownToMrkdwn(result.text);
  if (text.length > 3000) {
    await app.client.files.uploadV2({
      channel_id: event.channel,
      content: result.text,
      filename: 'response.md',
      thread_ts,
    });
  } else {
    await app.client.chat.update({
      channel: event.channel, ts: thinking.ts, text
    });
  }
});

05 — Claude Code SDK Integration

The Brain

The Claude Code SDK provides the same agent loop as the CLI — tool use, file editing, bash execution — but with structured programmatic I/O. It's the core of the entire system.

SDK Invocation

Each task triggers a Claude Code SDK call with:

  • prompt — the user message or scheduled task description
  • workingDirectory — always /workspace
  • sessionId — for conversation continuity
  • systemPrompt — injected memory context
  • allowedTools — read, write, edit, bash, MCP tools
Structured Results

The SDK returns structured data, not a text blob:

  • Text responses — the main answer
  • Tool calls — file reads, edits, bash commands
  • Tool results — command output, file contents
  • Session ID — for resuming later

This lets you decide what to surface in Slack: just the text, tool summaries, or full output.

Critical: Task Queue with Concurrency 1

Max plan rate limits are per-account, not per-device. If a cron task and a Slack request hit Claude Code simultaneously, one will fail. Use p-queue with concurrency: 1 and priority levels: interactive requests (priority 10) always jump ahead of scheduled tasks (priority 1).

06 — MCP Servers

External Service Access

MCP servers give Claude Code access to Gmail, Fireflies, Calendar, and more. These are self-hosted — not the same as claude.ai's built-in integrations.

Important Distinction

The Gmail, Fireflies, and Calendar integrations in claude.ai (the web app) are hosted by Anthropic with managed OAuth. They are not portable to Claude Code CLI. You need to run your own MCP servers that wrap these APIs directly.

07 — Persistent Memory

The Long-Term Brain

A structured file system that Claude Code reads and writes natively. No database, no adapter — just files. Git-tracked for version history, human-editable via SSH, self-maintaining via CLAUDE.md instructions.

/workspace/memory/ ├── CLAUDE.md ← Memory management rules ├── me.md ← Identity, role, preferences │ ├── clients/ │ ├── _index.md ← Client list + status │ ├── acme-corp.md ← Per-client history │ ├── globex.md │ └── vip.md ← Priority monitoring list │ ├── projects/ │ ├── _index.md │ └── website-redesign.md │ ├── todos.md ← Master to-do list ├── decisions.md ← Decision log + rationale │ ├── weekly/ │ ├── 2026-W11.md ← Auto-generated summaries │ └── 2026-W12.md │ └── learned-context.md ← "Remember X" from Slack
Why Files Over a Database?
  • Native I/O: Claude Code reads/writes files already — zero adapter code
  • Human-editable: SSH in and edit directly when needed
  • Git-trackable: Full version history of your memory for free
  • Self-organizing: Claude Code can create new files, reorganize, archive stale info
  • Transparent: You can always see exactly what the bot "knows"
Self-Maintenance via CLAUDE.md

The root CLAUDE.md instructs Claude Code to update memory after every task:

  • New client info → update their file
  • New to-dos → add to todos.md
  • Key decision → log in decisions.md
  • Completed to-do → mark done with date
/workspace/memory/CLAUDE.md — Memory management rules
# Memory System

Read all files in this directory tree for context before any task.

## After Every Task
1. If you learned new info about a client, update their file in clients/
2. If new to-dos emerged, add them to todos.md with due dates
3. If a to-do was completed, mark it done with the completion date
4. If a key decision was made, log it in decisions.md with rationale

## "Remember" Commands
When the user says "remember X", append to learned-context.md.

## Weekly Summaries
Every Friday, create a weekly summary in weekly/ covering:
- Client activity and changes
- Completed and new to-dos
- Key decisions made
- Items needing follow-up next week

## Maintenance
Keep files concise. Archive stale info to an archive/ directory.
Do not delete — archive. Memory is append-mostly.
08 — Scheduler & Proactivity

Working While You Sleep

Scheduled tasks make the bot proactive — it monitors, alerts, and prepares without being asked. The key principle: silent when nothing to report.

Schedule Task Behavior Posts to Slack?
0 8 * * 1-5 Morning Briefing Check emails, today's meetings, overdue to-dos. Synthesize into a summary. Always
*/15 * * * * Urgent Email Monitor Check for VIP client emails. Only alert if time-sensitive. Only if urgent
0 * * * * Meeting Prep Check calendar for meetings in next 2 hours. Post prep brief 30 min before. If meeting found
0 9,13,17 * * * Deadline Watchdog Review todos.md for items due within 48 hours. If deadlines found
0 16 * * 5 Weekly Review Summarize week's activity. Update weekly/ files. Post to Slack. Always
src/scheduler.ts — Proactive task definitions
import cron from 'node-cron';
import { runTask } from './queue';

// Morning briefing — weekdays at 8am
cron.schedule('0 8 * * 1-5', () => {
  runTask({
    prompt: `Check emails and Fireflies transcripts from the last 24 hours.
      Review today's calendar. Check todos.md for overdue items.
      Post a concise morning briefing to Slack.`,
    priority: 1,  // low priority — interactive requests jump ahead
  });
});

// Urgent email monitor — every 15 minutes
cron.schedule('*/15 * * * *', () => {
  runTask({
    prompt: `Check for new emails from VIP clients (see /memory/clients/vip.md).
      If anything looks urgent or time-sensitive, alert in Slack.
      If nothing urgent, do NOT post anything.`,
    priority: 0,  // lowest — never blocks interactive work
    silent: true,  // suppress "working on it" indicator
  });
});

// Weekly review — Friday at 4pm
cron.schedule('0 16 * * 5', () => {
  runTask({
    prompt: `Review all client and project activity this week.
      Create a weekly summary in /memory/weekly/.
      Post the summary to Slack.`,
    priority: 1,
  });
});
Design Principle: Silent When Nothing to Report

A bot that posts noise every 15 minutes will get muted. Every proactive task prompt explicitly says "if nothing important, don't post." The bot should only speak up when something matters. This is what makes the difference between an assistant and an annoyance.

09 — Data Flows

Interactive & Proactive Paths

Interactive Flow
  1. User @mentions bot in Slack
  2. Bot posts "working..." placeholder
  3. Message queued Priority 10
  4. Claude Code SDK invoked with prompt + workspace
  5. Reads CLAUDE.md + memory for context
  6. Uses tools (files, bash, MCP) as needed
  7. Updates memory files if new info learned
  8. Result formatted → placeholder replaced
Proactive Flow
  1. Scheduler triggers at configured time
  2. Task queued Priority 1
  3. Waits for any interactive tasks to finish
  4. Claude Code SDK invoked with task prompt
  5. Checks external services via MCP, reads memory
  6. If actionable → posts to Slack channel
  7. If nothing to report → silent, no post
  8. Memory files updated if needed

12 — Security & Edge Cases

Things to Watch

Bash Access is Powerful

Claude Code can execute arbitrary shell commands. For a personal single-user system this is acceptable, but be aware: anything your VM user can do, Claude Code can do. Run as a non-root user with limited permissions.

Sensitive Data in Memory

Memory files will contain business context — client names, deal values, to-dos. Keep the VM access locked down. If you git-push memory to a remote, use a private repository. Never store API keys or tokens in memory files — use env vars.

Token Expiry Handling

Max OAuth tokens will expire. The bot must catch auth errors, pause all queued tasks, and alert you in Slack. Implement a simple health check that runs before each SDK invocation and short-circuits if credentials are invalid.

Rate Limit Awareness

Max plan has usage rate limits that are per-account. Running many proactive tasks can eat into your interactive quota. The task queue (concurrency: 1, with priority) prevents collisions, but monitor your usage patterns and adjust cron frequencies if needed.
